Background

This journal system exists to solve one problem: development sessions are irregular and often end abruptly, making re-entry disorienting. Without a record, the story of what was built, why decisions were made, and what comes next lives only in chat conversations — which don't persist between sessions.

The system captures two things:

  • DEVLOG.md — a running narrative log of sessions: what happened, what broke, what was learned, notable moments. This is also the raw source material for content (LinkedIn, blog posts, YouTube Shorts on "Homeless Builder", eventually a book and course).
  • CURRENT.md — a per-project status snapshot, overwritten each session, so returning to a project after any gap starts with clear context rather than confusion.

The design principle throughout: the human stays in the loop. Nothing is written permanently without a confirmation step (except the manual-entry form, which is your own direct account and skips that step deliberately — see Manual Entry).

File Structure

~/claude-journal/
  CLAUDE.md                     — librarian session instructions for Claude Code
  CURRENT.md                    — live project status (overwritten each session)
  DEVLOG.md                     — running narrative log (newest entries at top)
  devlog.html                   — browser viewer (no build step required)
  new-entry.html                — manual entry form (writes via journal_server.py)
  journal_server.py             — local write-enabled server for the above
  .claude/
    commands/
      log.md                    — /log slash command definition

~/.local/bin/
  log-session                   — run from any project to capture a session
  read-log                      — terminal viewer for DEVLOG.md

How the System Works

Moving parts

log-session is the script you run at the end of a work session. It gathers recent git commits, calls Claude Code in headless mode to summarize the session, pauses for your confirmation, checks for stray .md files outside known project folders, then hands the confirmed summary to the journal's /log command.

The journal Claude Code session (run inside ~/claude-journal/) receives the handoff, drafts the DEVLOG entry and CURRENT.md update, shows you both, and commits on your confirmation.

new-entry.html + journal_server.py is a direct manual path: a form you fill out yourself, validated and sanitized, written straight to DEVLOG.md with no AI involved and no review step, since it's already your own first-hand account.

read-log and devlog.html are for reading — terminal and browser respectively.

What it cannot capture automatically

Claude Code in headless mode only sees files and git state, not chat conversations. Decisions, reasoning, and pivots that happened in a Claude.ai chat session need to be captured manually, either through the form or by writing directly.

Normal Use — Step by Step

At the end of a git-based project session

log-session ~/my-eleventy-sdai-h

Or from inside the project folder, with no argument:

cd ~/my-eleventy-sdai-h
log-session

Claude gathers recent commits, summarizes the session, shows you the draft, and hands it to /log on confirmation.

At the end of a conversation-only or non-git session

Use --manual for overseer sessions, planning sessions, or any work that happened entirely in conversation with no code commits:

log-session --manual overseer
log-session --manual ~/devlog-engine
log-session --manual

This opens your $EDITOR with a pre-filled handoff template. Fill in the fields, save, and close — the script sends it to /log.

Committing the entry

cd ~/claude-journal
git add DEVLOG.md CURRENT.md
git commit -m "Session log: YYYY-MM-DD — short description"

At the start of a new session

read-log --current     # check where you left off
read-log                # last 3 entries
read-log 5              # last N entries
read-log --all          # everything

Browser viewer and entry form

python3 ~/claude-journal/journal_server.py

Then open:

  • http://localhost:4242/devlog.html — read the log, filter by tag
  • http://localhost:4242/new-entry.html — write a manual entry directly

Manual Entry

Option A — the form

Open new-entry.html while journal_server.py is running. Fill it out, press Commit entry. It writes straight to DEVLOG.md. There is no review step, since this is your own first-hand account, not an AI-generated summary — but the server still validates and sanitizes input so stray formatting characters can't corrupt the file.

Option B — write it yourself

Open DEVLOG.md directly and insert a new entry below the header, above existing entries:

## YYYY-MM-DD — [short, plain-language title]

**Project:** [project name]
**Tags:** [Technical] [Process] [Story]

**What happened:**
[2-4 sentences, plain language]

**Obstacles hit:**
[Real problems and how they were resolved. Omit if nothing went wrong.]

**What was learned:**
[The transferable lesson. Omit if nothing notable.]

**Outcome:**
[Concrete state at session end]

**Related commits:**
[Commit hashes if applicable. Omit if none.]

---

Option C — the journal Claude Code session

Open a Claude Code session inside ~/claude-journal/ and run /log, then paste your notes or a context fragment and let it draft the entry.

Entry Writing Guidelines

  • Story and reasoning, not code changes. Git records the precise what. DEVLOG records the why. "The font felt too busy and was toned down" is right. "Changed font-size from 2.8rem to 2.4rem" belongs in a commit message.
  • Honest, plain language. No marketing tone. This is source material for content that works because it's real.
  • Capture genuine friction. Don't clean up what went wrong — that's often the most useful part.
  • Quote-worthy moments verbatim. If something was said in a way worth preserving exactly, write it exactly.
  • Keep it tight. A few solid paragraphs, not a transcript.

Tags

TagUse for
[Technical]A specific tool, setup, or implementation lesson
[Process]How the work was approached, AI collaboration pattern, workflow decision
[Story]Frustration, reframe, realization, human moment worth capturing for content

Reading the Log

read-log              # last 3 entries (default)
read-log 5            # last N entries
read-log --all        # full log
read-log --current    # CURRENT.md project status only

The browser viewer shows CURRENT.md status at the top, then the full log with tag filtering.

Troubleshooting

log-session fails with "No such file or directory"

The journal directory path in the script is wrong. Check this line near the top of ~/.local/bin/log-session:

JOURNAL_DIR="$HOME/claude-journal"

The form shows "Could not reach the server"

journal_server.py isn't running. Start it with python3 ~/claude-journal/journal_server.py and leave that terminal open while you use the form.

The browser viewer shows "Could not load DEVLOG.md"

You're opening the file directly rather than through the server. It needs to be served, not opened as a raw file path.

read-log command not found

cp /path/to/read-log ~/.local/bin/read-log
chmod +x ~/.local/bin/read-log
echo $PATH | grep local   # confirm ~/.local/bin is on PATH

git shows "nothing to commit" when DEVLOG.md was edited

git rm --cached DEVLOG.md
git add DEVLOG.md
git status

log-session warns about an unrecognized project directory

Add the new project to the array near the top of ~/.local/bin/log-session:

KNOWN_PROJECT_DIRS=("$HOME/my-eleventy-sdai-h" "$HOME/claude-journal")

Known Limitations

  • Git-mode summarization only sees git history and file state, not chat conversations. Use --manual or the form for conversation-only sessions.
  • Stray .md detection looks only one day back and at shallow directory depth.
  • No multi-project logging in one pass yet — run log-session separately per project.
  • Git-mode summary quality depends on commit message quality.
  • journal_server.py is local-only by design (binds to 127.0.0.1). It does not run git commands — writing and committing remain separate steps.
  • The home directory (~/) has no git repo — always use log-session --manual overseer for overseer sessions.
